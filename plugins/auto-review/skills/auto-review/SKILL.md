# セルフレビュー

現在のブランチの変更差分に対して、複数のレビュアーによる並列レビュー → 批判的評価 → 自動修正を実行する。

## 実行フロー

### ステップ1: 準備

1. このスキルと同じディレクトリにある `reviewers.yml` を読み込み、レビュアー一覧を取得する
2. `git diff` でワーキングツリーの変更差分を取得する（差分がない場合は `git diff origin/main...HEAD` でブランチ差分を取得）
3. 差分がなければ「レビュー対象の変更がありません」と報告して終了する

### ステップ2: 並列レビュー

`reviewers.yml` に定義された各レビュアーを、Agentツールで**1つのメッセージで並列に起動**する。

各レビュアーエージェントには以下を渡す：
- レビュアーの `agent` フィールドが指すファイルの内容（エージェント定義）
- `references/report-format.md` の共通レポートフォーマット（codex-reviewer を除く）
- レビュー対象の差分
- 「修正は行わず、レビュー指摘の報告のみ行うこと」という制約
- レビュアー名（レポートの識別に使用）

**重要**: codex-reviewer は `codex` コマンドをBashで実行してレビューする特殊なレビュアーである。エージェント定義の指示に従って実行し、共通レポートフォーマットは渡さない（codex の出力をそのまま使う）。

### ステップ3: 批判的評価と自動修正

全レビュアーの結果が揃ったら、以下の手順で批判的評価を行う。

#### 3-1. レビュー指摘の一覧化

全レビュアーの指摘を統合し、一覧化する。複数レビュアーが同じ指摘をしている場合は重複としてマークする（重複は信頼度が高い指摘として扱う）。

#### 3-2. 各指摘の妥当性評価

レビュアーの指摘を鵜呑みにせず、以下の基準で技術的妥当性を独自に判断する：

- **技術的に正しいか**: 指摘の内容が技術的に妥当か
- **プロジェクトの方針・コンテキストと合致するか**: CLAUDE.md や既存のコードベースの方針に沿っているか
- **実装上のトレードオフとして妥当か**: 指摘された点が意図的な設計判断である可能性はないか
- **修正によって新たな問題が発生しないか**: 修正の副作用を考慮する
- **重複指摘**: 複数レビュアーが同じ点を指摘している場合、その指摘の信頼度は高い

#### 3-3. 妥当な指摘の修正

妥当と判断した指摘のみ、コードを修正する。

### ステップ4: レビュー対応サマリー

修正完了後、以下の形式でサマリーを出力する。
サマリーの目的は「修正判断が適切だったかをユーザーがダブルチェックできること」であり、ユーザーが流し読みしただけで **何が問題で、どう直したか / なぜ直さなかったか** を把握できるように書く。元の指摘内容・判断根拠を省略しすぎないこと。

#### 記載ルール

各指摘について以下を必ず記載する：

1. **見出し**: 見出し行は「[Round N] レビュアー識別子」のみとし、**指摘の要点は直下の別行に太字で書く**（長いレビュアー識別子と指摘要点が1行に詰まらないようにするため）
   - レビュアー識別子は reviewers.yml の name をそのまま使う（例：general-reviewer #1, simplify-reviewer #3, codex-reviewer #1）
   - 複数レビュアーが同じ指摘をしている場合は「general-reviewer #5 / simplify-reviewer #3」のようにスラッシュで併記する
   - レビューループを複数回実施した場合は先頭に `[Round N]` を付ける（1回のみなら省略可）
   - 見出し行の下に1行空行を入れ、`**指摘の要点を一言で**` を置く
2. **指摘内容** / **修正内容** / **対応しない理由**
   - ラベル（`- **指摘内容**:`）の直後で改行し、**空行を1行挟んでから**インデント付きで本文を書く（Markdown のレンダリング上、空行がないと1行に繋がってしまうため）
   - 本文は**2〜3文で具体的に書く**。1行に詰め込まない
   - 該当ファイル・関数・変数名など、コードを特定できる固有名詞を含める
   - 「指摘内容」は「何がどういう理由で問題なのか」が単独で読んでわかるように書く
   - 「修正内容」は「どのファイル / 何を / どう変えたか」を書く。副次的に解消された問題があれば併記する
   - 「対応しない理由」は「なぜ対応不要と判断したか」を書く。意図的な設計判断・スコープ外・別PR対応など、判断根拠を明示する

#### フォーマットのポイント

- 各指摘の間には**必ず空行**を入れて視覚的に区切る
- 見出し（####）の後と、箇条書きの前後にも空行を入れる
- ラベル（`- **指摘内容**:`）の直後で改行し、**空行を1行挟んでから**本文を次の行にインデント付き（半角スペース2つ）で書く。1行に詰め込まない
- 長い説明は途中で読点や改行を入れて折り返しやすくする
- レビューループが複数回実施された場合は冒頭に **「レビューループ: N回実施（終了理由: ...）」** の1行サマリーを入れる

#### 出力テンプレート

```markdown
## レビュー対応サマリー

レビューループ: N回実施（終了理由: ...）

### 対応した指摘

#### [Round 1] general-reviewer #1 / simplify-reviewer #2

**[指摘の要点を一言で]**

- **指摘内容**:

  [該当箇所（ファイル/関数名）と、レビュアーが何をどう問題視したかを2〜3文で。
  背景やリスクも含める]

- **修正内容**:

  [どのファイル/何を/どう変えたかを2〜3文で。
  副次的に解消された他の指摘があれば併記]

#### [Round 1] general-reviewer #3

**[指摘の要点]**

- **指摘内容**:

  [2〜3文で具体的に]

- **修正内容**:

  [2〜3文で具体的に]

### 対応しなかった指摘

#### [Round 1] codex-reviewer #1

**[指摘の要点]**

- **指摘内容**:

  [2〜3文で具体的に]

- **対応しない理由**:

  [意図的な設計判断 / スコープ外 / 別PR対応 など、判断根拠を2〜3文で]
```

#### 良い例 / 悪い例

**悪い例**（見出しに識別子と要点が詰まっている、本文も1行で背景が不明）:

```markdown
#### general-reviewer #1: RAF cleanup 順序問題
- **指摘内容**: tick() で RAF cleanup の順序がおかしい
- **修正内容**: cancelled フラグを追加した
```

**良い例**（見出しはレビュアー識別子のみ、要点は直下の太字行、各ラベルの後に空行を入れて本文を別段落にする）:

```markdown
#### [Round 1] general-reviewer #1

**useBubblePosition の RAF cleanup 順序問題**

- **指摘内容**:

  useBubblePosition hook の tick() は先頭で次フレームの requestAnimationFrame を登録する実装。
  unmount 直後にも1回余計に tick が走る可能性があり、visible の高速トグル時に競合リスクがある。

- **修正内容**:

  cancelled フラグを導入し、cleanup で cancelled=true を立てた上で cancelAnimationFrame。
  tick 冒頭で cancelled チェックを追加。同パターンを useOverlayState にも適用した。
```
